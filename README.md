# エンジニア向け英語学習アプリ 企画・計画書

> **コンセプト: 「今こそ、エンジニアは英語を勉強しましょう」**
> 毎日5分、AIと一緒に「実務で使える技術英語」を積み上げる学習アプリ。

---

## 1. プロダクトビジョン

### ワンライナー
**「コードを書くように、英語を書けるエンジニアになる。1日5分、AIが伴走する技術英語特化アプリ」**

### ビジョン
世界中のドキュメント・OSS・Issueを「翻訳を挟まずに」読み書きできるエンジニアを増やす。エンジニアにとっての英語は教養ではなく、キャリアの可動域そのものであることを当たり前にする。

### ミッション
- 英語学習の「挫折ポイント」である継続と実務との乖離を、AIと5分設計で解決する
- 学んだ英語がその日のうちにコミットメッセージやPRコメントで使えるように、学習コンテンツを実務直結にする

---

## 2. 目的・背景 — なぜ今、エンジニアに英語か

### 背景（2026年時点の環境変化）
1. **AI時代に一次情報へのアクセス速度が競争力になった**
   - LLM公式ドキュメント、モデルカード、論文、GitHub Issuesはほぼ全て英語
   - 日本語翻訳を待っていたら半年遅れる
2. **リモート×グローバル採用の常態化**
   - 海外企業の日本からの採用が増加。英語での面接・Slack・PRレビューは日常
   - 年収ベースで国内比1.5〜3倍の差が現実に存在
3. **「技術はあるが英語で詰まる」エンジニアの層が厚い**
   - リーディングは何とかなるがライティング・スピーキングが弱い層が多数
   - 汎用英語アプリは文脈が仕事と乖離しており継続しない

### なぜ5分か
- 忙しいエンジニアの「歯磨きと同じ位置」に学習習慣を差し込むため
- 認知科学的にも分散学習（spaced repetition）×短時間は長時間一括より定着率が高い
- 毎日5分なら「やらない理由」を作れない

### なぜAIか
- マンツーマン英会話の費用・時間ハードルをLLMが破壊した
- 個人の弱点・業務ドメイン（Web/インフラ/ML/モバイル）に合わせたパーソナライズがLLMなら安価に可能
- 技術英語の「その表現、PRで失礼にならない？」という機微まで扱える

## 3. ターゲットユーザーペルソナ

### ペルソナA: 「情報の遅延に焦る」中堅バックエンドエンジニア
- 29歳男性、フリーランスor自社開発SaaS勤務
- TOEIC 600点台、技術ドキュメントは気合いで読む
- 悩み: 最新のLLM論文やOSS Issueを読むのに時間がかかる。海外カンファレンス動画が聞き取れない
- 欲求: 「訳さずに読める」感覚を取り戻したい
- 学習時間: 通勤中・休憩中の5-10分

### ペルソナB: 「転職準備中」の若手フルスタック
- 26歳、自社開発企業勤務3年目
- 外資/海外リモート転職を視野に入れ始めた
- 悩み: PRコメントで失礼な英語を書いていないか不安。技術面接の英語が怖い
- 欲求: 実際のPRやSlackで使える定型表現と、面接でつまずかない表現力
- 学習時間: 朝の5分

### ペルソナC: 「OSSコントリビュートしたい」ベテラン
- 38歳男性、SESまたは社内SE
- 技術力は高いが英語への苦手意識で一歩踏み出せない
- 悩み: OSSのIssueを立てたいが英語で書く勇気が出ない
- 欲求: 「このIssue文面、失礼じゃない？」をAIに確認してもらいたい
- 学習時間: 夜の寝る前5分

## 4. コア機能（MVP）

### MVP機能リスト（Phase 1で実装）
| # | 機能名 | 概要 | 優先度 |
|---|--------|------|--------|
| F1 | **Daily 5 Lesson** | 毎日届く5分完結レッスン。1レッスン=読む/書く/聞くの3ステップ | S |
| F2 | **技術英語フレーズカード** | コミットメッセージ・PRコメント・Slack定型句の暗記カード（SRS採用） | S |
| F3 | **AI Rewrite（PR英語添削）** | 書いた英文をLLMが「技術現場で自然な表現」に添削＋理由説明 | S |
| F4 | **実ドキュメント読解** | React/Next.js/Azure等の公式ドキュメント抜粋を使った精読問題 | A |
| F5 | **連続学習ストリーク** | Duolingo風の継続トラッキング（心理的報酬） | A |
| F6 | **ドメイン選択** | Web / インフラ / ML / モバイルから自分の分野を選び出題をチューニング | A |

### Phase 2 以降の拡張候補
- **AI会話練習**: 音声入出力で技術面接シミュレーション（Workers AI Whisper STT + LLM via AI Gateway）
- **GitHub連携**: 自分の実PRをアプリに取り込み添削
- **チーム機能**: 社内でスコアを競う（法人展開の種）
- **Podcast連動**: 技術系英語Podcastのディクテーション
- **オフライン対応**（モバイル）

## 5. 差別化戦略

### 競合マップ（2026年4月時点 調査済み）

| サービス | エンジニア特化 | AI会話 | 5分習慣設計 | 備考 |
|---------|-------------|--------|-----------|------|
| Duolingo Max | ✗ | ◎ (GPT-4o) | ◎ | $168/年。技術文脈ゼロ |
| Speak | ✗ | ◎ (hands-free) | ○ | $99〜180/年。会話特化 |
| EnglishCentral | △ | ○ | ✗ | 月2,000〜5,000円。習慣設計なし |
| Bizmates | ✗ | △ (2026/4統合) | ✗ | 月14,850円〜。法人向け |
| **エンジリッシュ** | **◎** | **✗** | **△** | **2017年開始、7年以上更新停止** |
| Speak Tech English | ◎ | ✗ | ✗ | テキスト教材中心 |
| **本アプリ** | **◎** | **◎** | **◎** | **この象限に競合なし** |

### 核心的発見
**「エンジニア × 毎日5分 × AI会話 × 技術英語特化」の象限は2026年時点で完全な空白地帯。**

エンジリッシュが唯一の先行例だが、2017年から実質更新停止。需要がないのではなく、**プロダクト投資が入らなかっただけ**という解釈が成り立つ。

### 我々の差別化ポイント（3本柱）
1. **業務シナリオ特化のAI会話**: PRコメント・Slack英語・技術面接・Issue投稿など「今日の仕事で使うシーン」でAIと練習
2. **Duolingo式ストリーク × 技術英語**: 5分習慣設計の実績ある手法に、エンジニア文脈のコンテンツを乗せる
3. **コミット・README・Issue英語をコアコンテンツに**: 他サービスが触れないエンジニア文化の英語表現を中核に据える

### 「なぜChatGPTで代替されないか」への回答
- ChatGPTは万能だが「学習設計」がない。何を練習すべきか自分で設計できない
- 本アプリは「カリキュラム×SRS×実務テンプレ」をパッケージ化し、5分に最適化されている
- 汎用LLMには「継続を支える仕組み（ストリーク・進捗可視化）」がない

## 6. 技術スタック

### 採用スタック: Cloudflare All-in-One（**採用**）

> **方針: 極力Cloudflareに閉じる。外部依存を最小化。**

| レイヤー | 採用技術 | Cloudflareサービス |
|---------|---------|-----------------|
| フロント | Next.js 15 (App Router) + Tailwind CSS | **Cloudflare Pages** |
| API | Hono on Workers | **Cloudflare Workers** |
| DB | Drizzle ORM + D1 (SQLite) | **Cloudflare D1** |
| KV / セッション | SRSスコア・ストリーク・一時キャッシュ | **Cloudflare KV** |
| ファイルストレージ | 音声・画像・コンテンツJSON | **Cloudflare R2** |
| LLM ルーティング | 外部プロバイダへのプロキシ・レート制限・コスト管理 | **Cloudflare AI Gateway** |
| LLM 実行（軽量） | Workers AI (Llama 3.3 70B / Mistral) — PoC・コスト最適化用 | **Cloudflare Workers AI** |
| LLM 実行（高品質） | OpenAI GPT-4.1 または Anthropic Claude — AI Gateway経由 | AI Gateway経由（Cloudflareに集約） |
| 音声STT（Phase2） | Workers AI Whisper (whisper-large-v3-turbo) | **Cloudflare Workers AI** |
| キュー / 非同期処理 | コンテンツ生成・バッチSRS更新 | **Cloudflare Queues** |
| 認証 | Workers + D1で自前JWT（OAuthはGitHub/Google） | Workers上で実装 |
| メール通知 | MailChannels（Cloudflare Workers無料統合） | **MailChannels via Workers** |
| CI/CD | Cloudflare Pages Git統合（push→自動デプロイ） | **Cloudflare Pages** |
| モニタリング | Workers Analytics + Logpush | **Cloudflare Analytics** |
| 決済 | Stripe（不可避の外部依存、これのみ） | — |

### Cloudflare外の依存（最小限）
| サービス | 理由 | 代替可否 |
|---------|------|---------|
| **Stripe** | 決済。代替なし | 将来Paddle等も検討可 |
| **OpenAI / Anthropic** | Workers AIで品質不足の場合のフォールバック | Workers AIが改善すれば段階的に移行 |
| **GitHub** | ソースコード管理 | Cloudflare Pagesは直接zipデプロイも可 |

### 採用理由
- **1プラットフォームで完結**: 課金先・設定先・ダッシュボードがCloudflare1箇所
- **コスト**: D1・KV・R2・Workers は無料枠が広い。個人開発フェーズはほぼ無料で回せる
- **エッジ実行**: Workers はグローバルエッジで低レイテンシ。LLMレスポンスのストリーミングも容易
- **AI Gatewayでプロバイダを抽象化**: OpenAI→Anthropic→Workers AIの切り替えをコード変更なしで行える
- **Next.js on Cloudflare Pages**: `@cloudflare/next-on-pages` で App Router 対応済み

### 却下した選択肢
- **Vercel + Supabase**: 課金先が分散する。Cloudflare一本化の方針に反する
- **Azure OpenAI直接利用**: AI Gatewayを挟む方がプロバイダロックインを避けられる
- **React Native + Expo (モバイルファースト)**: 初期はWebでPMFを見極めたい。モバイルはPhase3で検討

### 補助ツール
- **LLM評価**: 過去のrag-evalダッシュボード資産を流用し、プロンプト品質を数値化（Cloudflare Workers上で動作させる）

## 7. フェーズ分割

### Phase 1: MVP (2026-04 〜 2026-06)
**目的**: 「毎日5分回るプロダクト」を最小構成で作り、自分と友人10人で回す

- F1〜F3の実装（Daily 5 Lesson / フレーズカード / AI Rewrite）
- 30日分の初期コンテンツを手動作成（Web / インフラ / ML の3ドメイン × 10レッスン）
- ランディングページ公開
- **観察期間**: 2週間の自己ドッグフーディング → 継続率・挫折ポイント記録

**成功基準**:
- 自分が2週間毎日続けられる
- 友人5人以上が1週間継続
- AI添削の満足度アンケートで平均4.0/5.0以上

### Phase 2: 拡張と公開 (2026-07 〜 2026-09)
**目的**: 限定公開→一般公開。フリーミアム課金開始

- F4〜F6の実装（ドキュメント読解 / ストリーク / ドメイン選択）
- ユーザー登録・認証・決済統合
- プロダクトハント / Zenn / Bluesky / X での公開
- フリーミアム課金開始（無料: 3レッスン/日、有料: 無制限 + AI添削回数増）
- **観察期間**: 1ヶ月間、有料転換率と継続率を観測してから追加投資を判断

**成功基準**:
- MAU 500人
- 有料転換率 3%以上
- 週次継続率 40%以上

### Phase 3: AI会話・法人展開 (2026-10 〜 2027-03)
**目的**: AI会話機能と法人向けプランで収益を安定化

- AI会話練習（技術面接シミュレーション）
- GitHub連携（自分のPRを添削対象に）
- チームプラン（企業向け月額）
- 海外エンジニア市場への英語UI提供

**成功基準**:
- MAU 3,000人
- MRR 30万円
- 法人契約 3社

## 9. 収益モデル候補

### 選択肢A: フリーミアム + 月額サブスク（**採用**）
- 無料: 3レッスン/日、AI添削 3回/日まで
- Pro: 月額 980円 — 無制限レッスン、AI添削無制限、全ドメイン開放
- Team: 月額 500円/人（5人〜）— 法人向け、学習進捗ダッシュボード付き
- **採用理由**: 継続課金で開発費を賄いつつ、無料層でバイラル獲得

### 選択肢B: 買い切り（iOS/Android）
- 却下理由: 継続的なAPI費用（LLM）と相性が悪い

### 選択肢C: 広告モデル
- 却下理由: 学習体験を壊す。エンジニア層の広告耐性は低い

### 補助収益
- コンテンツ販売: 「技術英語フレーズ1000」などKindle出版
- アフィリエイト: 英語学習書籍・オンライン英会話

## 10. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| LLM API費用の暴走 | 高 | レート制限、キャッシュ、小モデルへのフォールバック、日次コスト上限 |
| ChatGPT等汎用AIに代替される | 高 | 学習設計・継続の仕組み・SRSで差別化。単発の添削ではなく体験全体で勝負 |
| コンテンツ作成コスト | 中 | LLMで初版生成→人間レビューのハイブリッド運用 |
| 継続ユーザーが集まらない | 高 | Phase 1の観察期間で早期に継続率を測定、目標未達なら機能ではなくオンボーディングから見直す |
| マスター自身のSES終了後の生活コスト | 中 | 6月末までに有料転換の感触を掴み、転職活動と並走 |
| 法的リスク（他社ドキュメント引用） | 中 | 引用は短文・出典明記。可能な限り自前コンテンツ＋OSS（MITライセンス等）を利用 |

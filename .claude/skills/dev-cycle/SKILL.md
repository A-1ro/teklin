# 開発サイクルスキル

対象 issue 番号: $ARGUMENTS

GitHub issue の開発サイクルを以下の手順で実行する。
**引数が指定されていない場合は、`gh issue list --state open --limit 1 --json number --jq '.[0].number'` を実行して最も番号が小さい open issue を自動選択する。**
基本的に自律的に進行し、ユーザーへの確認は重要な判断（要件の認識合わせ、設計方針の大きな分岐、レビューの確認事項）のみに限定する。
軽微な修正、検討推奨事項の対応判断、APPROVE 後のマージなどは確認なしで進める。

---

## 現在のリポジトリ情報

- リポジトリ: !`gh repo view --json nameWithOwner -q '.nameWithOwner' 2>/dev/null || echo '(不明)'`
- 現在のブランチ: !`git branch --show-current`
- ワーキングツリー: !`git status --short | head -5`

---

## フェーズ 1: Issue 読み込み・分析

1. 上記で確定した issue 番号に対し `gh issue view <番号>` で issue の詳細（タイトル、本文、ラベル、コメント）を取得する
2. issue の要件を以下の観点で整理する:
   - **ゴール**: 何を達成するのか
   - **スコープ**: 何が含まれ、何が含まれないか
   - **受け入れ条件**: どうなれば完了か
   - **技術的制約**: 使用すべき技術、避けるべきアプローチ
3. 不明点があればユーザーに確認する
4. 整理した要件をユーザーに提示し、認識が合っているか確認する

**→ 要件に不明点や大きな解釈の分岐がある場合のみユーザーに確認する。明確な場合はそのまま進む**

---

## フェーズ 2: 設計（必要な場合）

issue の規模と複雑さを判断し、設計が必要か決定する。

**設計が不要なケース** (即フェーズ 3 へ):
- バグ修正で原因と修正方針が明確
- 既存パターンの踏襲で済む小規模な変更
- ドキュメントやコメントの修正

**設計が必要なケース**:
- 新しい画面やコンポーネントの追加
- API エンドポイントの新規作成
- アーキテクチャに影響する変更
- 複数ファイルにまたがる変更

設計が必要な場合:
1. 影響を受ける既存コードを読んで理解する
2. 設計方針をユーザーに提示する:
   - 変更するファイルの一覧
   - 新規作成するファイルの一覧
   - データの流れ（フロント ↔ API ↔ DB/KV）
   - UI が関わる場合は画面構成の概要
3. 設計についてユーザーの承認を得る

**→ 設計方針に大きな分岐がある場合のみユーザーに確認する。明確な場合はそのまま進む**

---

## フェーズ 3: 実装

1. issue 用のブランチを作成する:
   ```
   git checkout -b feat/issue-<番号>-<短い説明> main
   ```
   ブランチ名は issue の内容に基づいて適切に命名する（feat/, fix/, docs/ 等）

2. @implementer エージェントに実装を委譲する。以下を明確に伝える:
   - フェーズ 1 で整理した要件
   - フェーズ 2 の設計方針（ある場合）
   - 実装の優先順位

3. UI の実装が含まれる場合は @designer エージェントにも委譲する

4. 実装完了後、以下を確認する:
   - `npm run typecheck` が通ること
   - `npm run lint` が通ること
   - 変更内容が要件を満たしていること

5. 変更をコミットする（意味のある単位で分割）

**→ 実装結果をユーザーに報告し、そのまま次のフェーズに進む**

---

## フェーズ 4: PR 作成

1. ブランチをリモートにプッシュする:
   ```
   git push -u origin HEAD
   ```

2. `gh pr create` で PR を作成する:
   - タイトル: 簡潔に変更内容を表す（70文字以内）
   - 本文: 以下のフォーマットで記述する

   ```
   ## Summary
   - 変更内容の箇条書き（1-3点）

   ## Related Issue
   Closes #<番号>

   ## Changes
   - 変更したファイルと変更の概要

   ## Test Plan
   - [ ] テスト項目のチェックリスト
   ```

3. PR の URL をユーザーに報告する

---

## フェーズ 5-6: レビュー → 修正ループ

以下を **APPROVE が出るまで繰り返す**:

### 5. レビュー

@reviewer エージェントにレビューを委譲する。以下を明確に伝える:
- フェーズ 1 の要件
- フェーズ 2 の設計方針（ある場合）
- PR の差分（`gh pr diff` の結果）

レビュー結果は必ず `gh pr comment` で PR にコメントとして残す。
「確認事項 ([QUESTION])」がある場合のみユーザーにも報告・確認する。それ以外はそのまま修正に進む。

### 6. レビュー修正

レビューで指摘された問題を分類する:

- **修正必須 ([SECURITY], [BUG], [DESIGN] の「問題」)**: @implementer / @designer に修正を委譲
- **検討推奨**: 自身で判断して対応するか決定する（判断が難しい場合のみユーザーに確認）
- **確認事項 ([QUESTION])**: ユーザーに回答を求める

修正完了後:
1. `npm run typecheck` と `npm run lint` を再実行
2. 修正内容をコミット
3. `git push` でリモートを更新
4. PR にレビュー修正の概要をコメントする:
   ```
   gh pr comment <pr-number> --body "レビュー修正内容の説明"
   ```

**→ 再びフェーズ 5 のレビューに戻る**

### APPROVE 判定基準

@reviewer から以下の条件を全て満たすレビュー結果が返された場合に APPROVE とする:
- 「問題 (修正必須)」セクションに項目がない
- 「確認事項」が全て解決済み
- セキュリティ上の懸念がない

APPROVE 時、そのままフェーズ 7・8 に進む（ユーザー確認不要）。

---

## フェーズ 7: 残タスクの Issue 化

レビュー過程で見つかった以下の項目を新しい issue として作成する:

- 「検討推奨」のうち、今回対応しなかったが将来対応すべき項目
- 実装中に発見した改善点やリファクタリング候補
- スコープ外だが関連する作業

各 issue は以下のフォーマットで作成する:
```
gh issue create --title "タイトル" --body "本文" --label "適切なラベル"
```

残タスクがない場合はこのフェーズをスキップする。
作成した issue の一覧をユーザーに報告する。

---

## フェーズ 8: PR マージ

1. APPROVE 済みであればユーザー確認なしでマージする:
   ```
   gh pr merge <pr-number> --squash --delete-branch
   ```
   ※ squash マージをデフォルトとする。ユーザーが別のマージ戦略を希望する場合は従う
3. ローカルブランチを main に戻す:
   ```
   git checkout main
   git pull origin main
   ```
4. 完了報告: マージされた PR の URL と、作成した関連 issue があればその一覧を表示する

---

## 中断・再開

- 各フェーズの開始時にタスクの進捗を更新する
- ユーザーが中断を要求した場合、現在のフェーズと進捗を報告して停止する
- 途中で問題が発生した場合、状況をユーザーに報告して指示を仰ぐ
